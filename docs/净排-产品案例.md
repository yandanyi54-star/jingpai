# 净排 · 产品案例（作品集）

> 一份给产品招聘官看的项目复盘——我如何从一个真实痛点出发，做出一个自己每天都在用的公众号排版工具。
> 当前版本 v0.14.2 ｜ 2026-07

## 一句话定位
**净排**是一个纯前端、不登录、文章不出本机的公众号 Markdown 排版工具——让用 Markdown 写作的人，一键得到可以直接粘进微信后台、且好看的排版。

---

## 一、背景与问题：公众号排版为什么是个痛点

我自己写公众号（起步期），每篇文章最耗神的不是写，是**排版**：

- **秀米 / 135 编辑器**：模板重、广告多、学习成本高，且不适合已经用 Markdown 写完的人。
- **mdnice（墨滴）**：功能最全，但**强制登录 + 文章被传到云端**——对把文章当核心资产的人来说，这是真实的心理门槛。
- 现有工具要么"重要"，要么"不隐私"，没有兼顾。

这个痛点不是我一个人的：很多个人号主既想要好看的排版，又不想把未发布的文章交给别人的服务器。

> 我的角色：**我既是这个产品的用户，也是它的建造者。** 这让我不需要"假设用户"，能直接用自己使用时的摩擦来驱动每一次决策。

---

## 二、用户与洞察

- **目标用户**：用 Markdown 写作的个人公众号作者；在意隐私与简洁；不想被登录墙和广告打扰。
- **关键洞察**：「文章不出本机」不是营销噱头，而是真实诉求——排版工具的输入是用户最值钱的内容，本地化处理能直接消除信任成本。
- **我是用户本人**：这意味着需求优先级由"我自己排版时卡在哪"决定，而不是由功能清单决定。

---

## 三、竞品分析（定位是怎么来的）

| 产品 | 强项 | 弱点 | 净排的机会 |
|------|------|------|-----------|
| mdnice | 主题多、图床、多平台 | 强制登录、文章上云 | 本地隐私 + 不登录 |
| Plainly | 本地化、界面现代 | 无 AI、装饰能力弱 | AI 自带 key + 原生装饰卡片 |
| 秀米 / 135 | 模板生态大 | 重、广告、不适合 MD | 轻量、Markdown 优先 |

→ 净排卡位：**本地隐私 + 自己掌控的 AI + 原生装饰**，做"一个写作者用得安心的排版工具"。

---

## 四、产品决策与权衡（这份工作的核心）

我做了几个主动取舍，每一个都写了清楚的理由——对产品岗来说，**为什么不做某件事**往往比做了什么更说明问题：

**决策 1：纯前端、不登录、不入库。**
- 代价：没有账号体系、没有云端同步、难以做留存与数据分析。
- 收益：隐私承诺可信（不是嘴上说）、零服务器成本、部署极简。
- 判断：对"信任"型产品，少一样东西比多一样功能更重要。

**决策 2：AI 用用户自带 key，浏览器直连厂商。**
- 代价：用户要自己填一个 key。
- 收益：我不碰用户数据、不背算力成本、没有合规风险，且"自带 key"本身就是隐私立场的一部分。

**决策 3：装饰元素（封面 / 分隔 / 金句）用纯 CSS，不用 SVG。**
- 代价：表现力受限。
- 收益：微信后台对内联样式兼容最稳，导出的文章文字可选、可复制，不会被微信吞掉。

**决策 4：导出恒定浅色，预览里加"微信深色"模拟开关。**
- 背景调研：微信的深色是**读者手机端算法反转**——无彩色按算法反、有彩色原样保留；文章本身无法控制、也不支持 `<style>`。
- 决策：导出不做"假适配"；模拟开关只用来**提前暴露低对比风险**给作者看。这体现"基于事实决策，而不是想当然"。

## 五、迭代记录：执行力与"不假装成功"

从 v0.13 到 v0.14，我做了一轮完整打磨，重点讲三个**被发现并修掉的"假成功" bug**（最能体现产品素养）：

1. 编辑器"字间距"滑杆拼错单位，只有预览生效、编辑器本身无效——用户看到滑杆却没反应。
2. "跟随系统"模式外壳被写死成深色，系统浅色时内外明暗割裂。
3. AI「扩写 / 改写 / 翻译选中文本」因定位方式错误**静默失败且无提示**——用户以为成功了。

→ 我的原则：**宁可在发布前暴露问题，也不能让用户在不知情下"以为成功了"。**

## 七、复盘与反思
